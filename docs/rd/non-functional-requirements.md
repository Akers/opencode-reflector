# 四、非功能需求

> 本文档是 [需求与设计方案主文档](../requirements-and-design.md) 的组成部分。

---

### NFR-001：性能

| 场景 | 指标 |
|------|------|
| 单会话（100 条消息）指标提取 | < 2 秒 |
| 单会话情感分析（含 20 条人类消息） | < 5 秒 |
| 日报生成（10 个会话） | < 10 秒 |
| 内存占用（峰值） | < 200 MB |
| 数据库单文件大小（1 年，日均 10 会话） | < 100 MB |

### NFR-002：可靠性

| 场景 | 策略 |
|------|------|
| 反思过程部分失败 | 已提取的指标正常保存，失败部分记录错误日志，标记为 `PARTIAL` 状态 |
| 情感分析模型不可用 | 跳过情感分析，对应指标标记为 `N/A`，使用 L3 关键词匹配降级 |
| 数据库写入失败 | 重试 3 次（指数退避），仍失败则写入 fallback JSON 文件 |
| 并发触发 | 使用文件锁（`.reflector/data/reflector.lock`）保证同一时间仅一个反思进程运行 |
| 日报幂等 | 同一日期的日报追加更新，不覆盖 |

### NFR-003：隐私与安全

| 要求 | 实现方式 |
|------|---------|
| 数据不离开本地 | 所有指标数据、情感分析结果仅存储在本地 `.reflector/` 目录 |
| 原始对话不存储 | 只提取指标数值和统计结果，不存储原始消息内容 |
| 敏感信息脱敏 | 送入情感分析模型前，自动移除 API Key、密码、Token 等敏感 pattern |
| 默认排除版本控制 | `.reflector/` 目录默认加入 `.gitignore` |
| 情感分析可禁用 | 提供配置项完全禁用情感分析功能 |
| 数据生命周期管理 | 提供清理命令自动删除过期数据 |

### NFR-004：可观测性

- **日志规范**：所有操作记录结构化日志（JSON 格式），包含时间戳、级别、模块、操作、耗时
- **日志输出**：`.reflector/logs/reflector.log`
- **关键事件日志**：
  - 触发事件（类型、时间）
  - 会话获取（数量、耗时）
  - 情感分析（会话 ID、耗时、Token 消耗）
  - 指标保存（会话 ID、指标数量）
  - 日报生成（日期、路径）
  - 错误与异常（详情、堆栈）
